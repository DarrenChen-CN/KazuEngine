# Review Checklist

> 每次 AI 交付 Nano-Feature 后，使用此清单进行 Review。
> 全部打勾 = 可以合并/继续；有任何未勾 = 需要修正。

---

## 代码规模

- [ ] 新增/修改代码总行数 < 300 行
- [ ] 涉及文件数 <= 5 个
- [ ] 如果超过限制，已拆分为多个 Nano-Feature

## 文档交付

- [ ] 提供了 What（50 字内功能描述）
- [ ] 提供了 Why（设计动机，100 字内）
- [ ] 提供了 Trade-offs（放弃的替代方案及原因）
- [ ] 提供了验证步骤（如何编译/运行/验证）

## 代码质量

- [ ] 所有 Vulkan 对象生命周期明确（谁创建、谁销毁、RAII 是否完备）
- [ ] 没有隐藏的 GPU 内存分配（显式使用 VMA 或自定义分配器）
- [ ] 错误处理路径覆盖（至少检查 `VK_SUCCESS`，使用 `VK_CHECK` 宏）
- [ ] 没有过度封装（原始 `Vk*` 句柄可通过 `handle()` 访问）
- [ ] Debug 模式下有合理的日志/assert（关键资源创建打印日志）

## 架构合规

- [ ] 只修改当前聚焦的层级，未跨层修改
- [ ] 公共接口符合 `docs/knowledge/` 中的契约约定
- [ ] 如果需要修改接口契约，已同步更新 knowledge 文档

## 可验证性

- [ ] 代码可以编译通过（提供了编译命令）
- [ ] 提供了运行验证方法（测试程序或主程序入口）
- [ ] 验证步骤中无 Validation Layer 报错

## 理解确认

- [ ] 我理解这段代码中每个 Vulkan API 调用的目的
- [ ] 我理解这个类/函数在整个架构中的位置
- [ ] 我知道如果这里出错了，应该如何调试
